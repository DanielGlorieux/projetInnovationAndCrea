## 📊 Artefacts SCRUM

### Definition of Ready (DoR)
✅ Une User Story est prête quand elle est :
- Claire et compréhensible
- Estimée en story points
- Validée par l'équipe
- Priorisée
- Indépendante ou dépendances identifiées

### Definition of Done (DoD)
✅ Une User Story est terminée quand :
- Le code fonctionne
- Les tests passent
- La revue de code est faite
- La documentation est à jour
- Le Product Owner valide

## 📅 Cérémonies SCRUM

### Sprint Planning (Début du sprint)
**Durée** : 2h pour un sprint de 2 semaines  
**Objectifs** :
- Définir le Sprint Goal
- Sélectionner les US du backlog
- Décomposer en tâches
- S'engager sur un objectif

### Daily Standup (Quotidien)
**Durée** : 15 min  
**Questions** :
- Qu'ai-je fait hier ?
- Que vais-je faire aujourd'hui ?
- Ai-je des blocages ?

### Sprint Review (Fin du sprint)
**Durée** : 1h  
**Objectifs** :
- Démontrer le travail accompli
- Recueillir le feedback
- Adapter le backlog

### Sprint Retrospective (Fin du sprint)
**Durée** : 1h  
**Objectifs** :
- Ce qui a bien fonctionné
- Ce qui peut être amélioré
- Actions concrètes pour le prochain sprint

---

## 📈 Métriques Clés

### Vélocité
Nombre de story points complétés par sprint  
**Objectif** : Stabilité et prédictibilité

### Burndown Chart
Points restants vs temps  
**Objectif** : Suivre l'avancement du sprint

### Cycle Time
Temps moyen pour compléter une US  
**Objectif** : Optimiser le flux

### Lead Time
Temps de la création à la complétion  
**Objectif** : Améliorer la réactivité

## 💡 Conseils et Bonnes Pratiques

### Rédaction de User Stories
```
En tant que [rôle]
Je veux [action]
Afin de [bénéfice]
```

**INVEST** :
- **I**ndépendante
- **N**égociable
- **V**alorisée
- **E**stimable
- **S**uffisamment petite (Small)
- **T**estable

### Estimation en Story Points
**Échelle de Fibonacci** : 1, 2, 3, 5, 8, 13, 21

- **1 point** : Trivial, < 2h
- **2 points** : Simple, 2-4h
- **3 points** : Moyen, 4h-1j
- **5 points** : Complexe, 1-2j
- **8 points** : Très complexe, 2-3j
- **13+ points** : À décomposer

### Priorisation
**Méthode MoSCoW** :
- **M**ust have : Critique
- **S**hould have : Important
- **C**ould have : Confort
- **W**on't have : Hors scope

## 🔍 Points d'Attention

### Erreurs Courantes à Éviter

❌ **Ne pas faire** :
- Changer le scope en cours de sprint
- Négliger le DoR/DoD
- Sous-estimer systématiquement
- Travailler sur trop de tâches en parallèle
- Ignorer les blockers

✅ **À faire** :
- Respecter les time-boxes
- Communiquer régulièrement
- Adapter et améliorer continuellement
- Célébrer les succès
- Apprendre des échecs
